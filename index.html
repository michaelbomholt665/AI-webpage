AI Rules for Handling "Locked" Files

What’s the Purpose?

The site’s goal is to let you build and simulate stories using AI logic (or at least fake it ‘til you make it!). It’s got sections like Simulators, Stories, Templates, Logs, and the big one: Storytelling. The project is designed to run entirely in the browser—no server-side dependencies like Node.js or http-server are needed, keeping it lightweight and accessible.

How’s It Supposed to Work?

  • Adding Entries: On the Storytelling page, you’ll find an “Input Feature” section. Paste some text, pick a category, and hit add—it’s that simple. Each entry gets its own little spot with options to expand/hide or delete it.
  • Managing Entries: Every entry has buttons:
    • Expand/Hide: Toggle the text to keep things tidy.
    • Delete This: Get rid of stuff you don’t need anymore.
  • Saving and Loading:
    • Entries stick around thanks to localStorage, so they’re there even if you close the browser.
    • Hit “Download Updated Library” to save everything as a library.json file.
    • Use “Load Library” in the top menu to upload a library.json file and update your entries.
    • Chat-Based Auto-Save (Expanded Details):
      • When you tweak this project in a chat with an AI (like Grok), it runs an “auto-save” in the background to protect your work.
      • It saves every time you change a file, such as index.html, script.js, or styles.css, capturing the full code state of all project files.
      • The AI keeps the last 10 full code states in its memory, allowing you to roll back to previous versions if needed.
      • To retrieve a saved state, say “load save X” (where X is 1 to 10) in the chat, and the AI will provide the code from that save point.
      • Use “list-auto-saves” to see up to 10 recent changes, with notes like “save nr1. ‘Added X to Y’” to track your progress.
      • Once 10 saves are reached, the AI exports a saves.json file automatically and clears the history to free resources, ensuring efficient memory usage.
      • You can also say “export-saves” to download a saves.json file at any time, which also clears the history.
      • The saves.json file has two sections: “latest_export_data” for the most recent export (up to 10 saves), and “previous_export_data” for past exports (limited to 5 previous exports, for a total of 50 save points).
      • In a new chat, provide the entire saves.json file and say “import-saves” to restore the history, allowing you to pick up where you left off.
      • Use the “Save Export Data” page to manage saves.json: load the file, review or edit the export data in a text field, and export a new saves.json with updated history.
      • Without exporting, saves reset per chat session—you won’t need to redefine this later.
      • Important Note: The auto-save feature in chat does not use JSON.parse due to compatibility issues with some project components. Instead, it relies on direct string manipulation for saving and loading states, ensuring reliability across all chat interactions.

It’s all browser-based—super lightweight and no fancy setup required.

Key Things to Know

  • Persistence: localStorage keeps your entries safe until you delete them or clear your browser’s storage.
  • File Format: When loading a library.json, make sure it’s got a storytelling key with an entries array inside, or it’ll choke. Check the code if you’re unsure.
  • Where’s the Magic?: Look in storytelling.html, script.js, and styles.css. They’re packed with comments to guide you through what’s what.

AI Rules for Handling "Locked" Files

Hey, future me! These rules are here to keep your finished files safe from accidental mess-ups while still letting you tweak them when needed. Think of "locked" files as your completed masterpieces—like stories.html—that shouldn’t change unless there’s a good reason. Here’s the breakdown:

  • What’s a "Locked" File?

    It’s a file you’ve finished and tested, like stories.html. It’s not set in stone (not read-only), but you should treat it with care and only edit it following these steps.

  • How to Edit a Locked File

    Before you touch it:

    • Say Why: Write down why you’re changing it (e.g., "The story list isn’t showing up—fixing a bug").
    • Keep a Note: Add a comment in the file or a changelog, like "2023-10-15 - Added missing div."
    • Check It Works: Test the file after editing to make sure it’s still doing its job.
  • Dealing with Shared Files (e.g., script.js or styles.css)

    These files are used by lots of pages, including locked ones, so be extra careful:

    • Flag the Important Bits: In the shared file, add comments like "This part is for stories.html—don’t break it!"
    • Look Before You Leap: Check those flagged parts before editing to avoid messing up locked files.
    • Test Everything: After changing a shared file, open up the locked files (like stories.html) and make sure they still work.
  • Keep Things Separate

    Move code specific to a locked file (e.g., JavaScript or CSS for stories.html) into its own files, like stories.js or stories.css, in the data folder. This way, shared files don’t mess with them.

  • Plan for Updates

    Write your code in chunks (like using ES6 modules) and add notes in shared files, like "Add stories.js back here if needed." This makes it easy to update later without breaking stuff.

  • Extra Safety (Optional)

    Use Git to save your work. If something goes wrong, you can roll back. Name your saves clearly, like "Split off stories.js from script.js."

Bottom line: These rules keep your project tidy and safe. Locked files stay solid, but you can still fix or improve them—just follow the steps. You’ve got this!
